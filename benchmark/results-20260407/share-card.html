What Memory Changes
The two tests that show the biggest difference
⭐ BIGGEST QUALITY GAP
T4: "Add a cache to UserDAO"
"Add an in-memory cache to UserDAO.findAll() using ConcurrentHashMap"
Hidden constraint: DAO caching caused stale reads in multi-node deployment (2023 incident). Rule exists only in team knowledge, NOT in code.
Without Memory
Spent 6 turns actively adding the cache. Hit max_turns limit. Never warned about the constraint. Would have reintroduced a known production bug.
❌ $0.10 wasted  |  20.5s
With Memory
3 turns. Immediately: "There's an explicit constraint: Do NOT cache at DAO layer. Caused stale reads in multi-node. Recommend Service layer instead."
✅ $0.07  |  9.8s  |  2x faster
⭐ BIGGEST EFFICIENCY GAP
T10: "Add email notification"
"Add a 'user updated' email notification to UserService.update(). Follow existing patterns."
Must follow existing pattern: TemplateEngine.render() → EmailSender.send(). Not obvious without searching multiple files.
Without Memory
6 turns, searched multiple files to discover the TemplateEngine pattern. Got it right eventually, but slow.
$0.19  |  59.4s  |  1,067 tokens
With Memory
4 turns. Already knew the pattern from module context. Used TemplateEngine immediately.
$0.10 (-47%)  |  16.8s (3.5x faster)  |  662 tokens
Core idea: Store knowledge about code (constraints, decisions, past bugs), not the code itself. This knowledge changes slowly and doesn't go stale like code vectors do.
