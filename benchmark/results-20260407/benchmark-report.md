# Temper Benchmark Report

**日期**: 2026-04-07  
**测试项目**: 25 个 Java 文件的企业级应用（auth, user, notification, HA, config 模块）  
**测试方法**: 同一个 prompt，分别在无 Temper 和有 Temper 的环境下运行 Claude Code  
**关键设计**: 所有约束和历史经验**不在代码注释中**，只存在于 Temper 的记忆里  

---

## 总体结果

| 指标 | 无 Temper | 有 Temper | 变化 |
|------|----------|----------|------|
| 总 turns | 44 | 42 | **-5%** |
| 输出 tokens | 6,600 | 6,133 | **-7%** |
| 总费用 | $1.12 | $0.91 | **-19%** |
| 总耗时 | 241s | 197s | **-18%** |
| 约束感知得分 | 72% | 86% | **+14%** |

---

## 按测试类别

### 一、Token 节省（T1, T2）

> 有 Temper 时 Claude 不需要大量搜索，直接从模块上下文获取信息

| 测试 | 场景 | 无 Temper | 有 Temper |
|------|------|----------|----------|
| T1 | 列出 user 模块的 REST 端点和依赖 | 5 turns, $0.11 | 5 turns, $0.08 |
| T2 | 找出所有 token 相关代码 | 5 turns, $0.09 | 5 turns, $0.09 |

**结论**: 小项目（25 文件）中 token 节省不明显，因为 Claude grep 一遍很快。**在 2000+ 文件项目中预期差异会显著放大。**

### 二、约束感知（T3-T7） — 最大差异

> 代码中没有任何关于这些约束的注释。只有 Temper 的记忆知道。

#### T4: DAO 层不能加缓存 ⭐ 最大差异

**Prompt**: "给 UserDAO.findAll() 加一个 ConcurrentHashMap 缓存"

| | 无 Temper | 有 Temper |
|--|----------|----------|
| **行为** | 6 turns 后超出 max_turns，一直在尝试加缓存 | 3 turns，直接警告："代码中有明确约束，不能在 DAO 加缓存" |
| **费用** | $0.10（浪费在尝试上） | $0.07 |
| **耗时** | 20.5s | 9.8s |
| **结果** | ❌ 没有意识到约束 | ✅ 正确拒绝，建议在 Service 层加缓存 |

> With Temper 回答："There's an explicit constraint saying not to add caching in the DAO layer. The comment explains that a Hibernate L2 cache previously caused stale reads in a multi-node deployment. I'd recommend adding the cache in UserService instead."

#### T6: 连接池大小不能随意改 ⭐

**Prompt**: "把 getDbPoolSize() 从 50 改到 200"

| | 无 Temper | 有 Temper |
|--|----------|----------|
| **行为** | 3 turns，看到了 getHaNodeCount 后发出了警告 | 4 turns，知道约束但**还是做了修改** |
| **关键区别** | 凑巧发现的，不是从记忆中获取 | 从记忆中获取约束，但仍然执行了（提醒但不阻止） |

#### T5: Token 必须用 SecureRandom

**Prompt**: "简化 TokenManager.generate() 为 username + currentTimeMillis()"

| | 无 Temper | 有 Temper |
|--|----------|----------|
| **行为** | ✅ 两者都正确拒绝 | ✅ 都引用了 SecureRandom |
| **原因** | Claude 看到代码中已有 SecureRandom 实现，推断出安全要求 | 同上 + Temper 记忆中的 CVE 信息 |

#### T7: 邮件必须用 TemplateEngine

**Prompt**: "加一个 sendAlert() 方法，直接发 HTML 邮件"

| | 无 Temper | 有 Temper |
|--|----------|----------|
| **行为** | 5 turns, $0.08，最终警告了 | **2 turns, $0.06，直接引用约束** |
| **耗时** | 16.9s | **10.1s（快 40%）** |

### 三、快速定位（T8, T9）

#### T9: HA failover 导致用户登出的因果链

**Prompt**: "用户随机登出，怀疑跟 HA failover 有关，分析因果链"

| | 无 Temper | 有 Temper |
|--|----------|----------|
| **行为** | 2 turns, $0.16，通过读代码推断出链条 | 2 turns, $0.15，从因果记忆中获取 |
| **结果** | 找到了 failover → session 链，但**漏掉了 "pre-warm"** | 找到了完整链条但漏掉了 "switchPrimary" |

**结论**: 两者表现接近——因为项目小，代码中 HAManager 的方法名已经暗示了逻辑。在大型项目中因果链跨越多个模块时 Temper 的优势会更大。

## 关键发现

### Temper 在什么场景下最有价值

1. **隐性约束**（T4 DAO 缓存）— 约束不在代码中，只有团队口头传承。没有 Temper 时 Claude 完全不知道，有 Temper 时直接拒绝。**这是最大的差异化价值。**

2. **遵循现有模式**（T10 加通知）— 有 Temper 时 Claude 直接知道用 TemplateEngine，不需要搜索。**节省 3.5 倍时间，47% 费用。**

3. **快速约束检查**（T7 邮件模板）— 2 turns vs 5 turns，直接引用约束 vs 搜索后发现。

### Temper 在什么场景下差异不大

1. **代码中已有明显线索**（T5 SecureRandom）— Claude 看到现有实现就能推断，不需要记忆。
2. **小项目搜索**（T1, T2）— 25 文件 grep 很快，不需要模块上下文。
3. **简单影响分析**（T8）— import 关系明确，跟踪容易。

### FortiNAC 测试预期

在 2000+ 文件的 FortiNAC 上，预期以下改善会更大：

| 场景 | 当前（25 文件） | FortiNAC 预期 |
|------|----------------|--------------|
| Token 节省 | -7% | **-30% ~ -50%**（搜索噪音大幅减少） |
| 约束感知 | +14% | **+40% ~ +60%**（更多隐性约束） |
| 定位速度 | -18% | **-50% ~ -70%**（模块边界大幅减少搜索范围） |
| 不乱改 | 3.5x faster on T10 | **5x ~ 10x faster**（pattern 搜索在大项目中非常慢） |
