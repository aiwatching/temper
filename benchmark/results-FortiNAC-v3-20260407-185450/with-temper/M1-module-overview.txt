## core_business_v2/ingestion Module Overview

The ingestion module is a **Spring Boot microservice** that processes network entity (host/user) ingestion events through a **Redis-stream-backed pipeline** with a **linear state machine**.

---

### REST Endpoints

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/api/v1/ingestion/events` | Synchronous event ingestion (60s timeout) |
| GET | `/api/v1/ingestion/health-check` | Health probe |
| POST | `/api/v1/entities/query` | Query failed entities with pagination |
| GET | `/api/v1/entities/{id}` | Get entity by ID |
| GET | `/api/v1/entities/{id}/display-status` | Get entity processing status |
| POST | `/api/v1/fingerprint/calculate` | Calculate attribute fingerprint |
| POST | `/api/v1/policy/apply-existing-entities` | Mark all entities for policy reprocessing |

---

### Request Processing Flow

```
REST Client / External System
        ↓
  ┌─────────────────────────────┐
  │  IngestionController        │  (or IngestionExternalConsumer for async)
  │  → publish to Redis stream  │
  │  → register callback future │
  └─────────┬───────────────────┘
            ↓
  ┌─────────────────────────────┐
  │  IngestionInternalConsumer  │  (Redis stream consumer group)
  │  → deserialize request      │
  │  → call orchestrator        │
  └─────────┬───────────────────┘
            ↓
  ┌─────────────────────────────┐
  │  IngestionOrchestrator      │  (routing logic)
  │  1. Redis cache lookup      │
  │  2. Creation lock acquire   │
  │  3. DB lookup               │
  │  4. Route to handler        │
  └─────────┬───────────────────┘
            ↓
  ┌─────────────────────────────┐
  │  FingerprintHandler         │  (Strategy pattern, 4 variants)
  │  Compare fingerprint + lock │
  │  → same attrs + lock:   noop│
  │  → same attrs + no lock: poll│
  │  → diff attrs + lock:  reset│
  │  → diff attrs + no lock: interrupt & retry│
  └─────────┬───────────────────┘
            ↓
  ┌─────────────────────────────┐
  │  EntityStateMachine         │  (linear 4-stage pipeline)
  │  PENDING → LABELED          │  (LabelService)
  │  LABELED → EVALUATED        │  (PolicyService)
  │  EVALUATED → ENFORCED       │  (EnforcementService)
  │  Each stage: 3 retries      │
  └─────────┬───────────────────┘
            ↓
  Callback completes → HTTP response
```

---

### Key Classes by Layer

**Controllers** — REST entry points:
- `IngestionController` — main ingest endpoint, publishes to Redis stream and waits on a `CompletableFuture` callback
- `EntityController` — entity query/status inspection
- `PolicyController` — triggers bulk reprocessing

**Consumers** — Redis stream listeners:
- `IngestionExternalConsumer` — receives from external stream, validates, forwards to internal stream
- `IngestionInternalConsumer` — core consumer, invokes the orchestrator, completes callbacks

**Orchestrator** — routing decision:
- `IngestionOrchestrator` — determines if entity is new/cached/in-DB, acquires creation locks, delegates to the right handler

**Handlers** (Strategy pattern via `FingerprintHandler` interface):
- `NewEntityHandler` — creates entity, runs state machine
- `CachedExistingEntityHandler` / `DbExistingEntityHandler` — computes fingerprint diff, selects one of 4 fingerprint handlers
- `DeviceTypeChangedHandler` — handles entity type mutation (e.g., HOST → USER_HOST_BINDING)
- `SameAttributesWithLock/NoLockHandler`, `DifferentAttributesWithLock/NoLockHandler` — the 4 fingerprint-case handlers

**State Machine**:
- `EntityStateMachine` — runs the `PENDING → LABELED → EVALUATED → ENFORCED` loop with per-stage retry (max 3), fail-point support, and interrupt checking
- `EntityStageTransactionRunner` — wraps stage transitions in DB transactions

**Stage Services** (business logic per stage):
- `LabelService` — executes labeling/rule engine (PENDING → LABELED)
- `PolicyService` — evaluates policy (LABELED → EVALUATED)
- `EnforcementService` — applies enforcement (EVALUATED → ENFORCED)

**Registries**:
- `IngestionCallbackRegistry` — maps requestId → `CompletableFuture` for sync REST responses
- `FingerprintHandlerRegistry` — maps `FingerprintCase` enum → handler instance
- `EntityProcessingRegistry` — tracks which entities are actively processing

**Recovery/Resilience**:
- `EntityRecoveryService` — recovers stuck entities and reclaimed pending messages
- `StuckEntityRecoveryJob` — scheduled scan for stuck entities
- `RedisDelayQueueUtil` / `DelayQueuePoller` — exponential backoff retry queue (2^n seconds)

---

### Key Design Patterns

1. **Dual-stream architecture**: External stream (untrusted input) → validation → Internal stream (processing). This separates ingestion from execution.

2. **Fingerprint-based deduplication**: Each request's attributes are hashed. If the fingerprint matches the existing entity and a lock is held, processing is skipped entirely (fast path).

3. **Distributed locking**: Two Redis locks per entity — a **creation lock** (prevent duplicate inserts) and a **process lock** (ensure only one state machine instance runs at a time). Both have 30s TTL.

4. **Interrupt-and-retry**: When a newer request arrives while a state machine is running, it sets an `interrupted` flag, waits for the lock, then restarts from PENDING with updated attributes.

5. **Version ordering**: Redis stream message IDs serve as version numbers — stale requests (lower version than last processed) are discarded.
