I'd push back on this change. The batched-transaction design here is **intentional and well-reasoned**, not inefficient. The code comments explain why:

**Line 266-270**: 
> "one transaction per update batch to avoid a huge long transaction"
> "if one batch fails, only that batch rolls back and already committed batches stay committed"

This is a deliberate design with clear benefits:

1. **Fault isolation** — If batch 5 of 100 fails, batches 1-4 are already committed and only batch 5 rolls back. A single transaction would discard all progress.
2. **Lock pressure** — One giant transaction holds row locks on potentially millions of rows for the entire duration. Batched transactions release locks between batches, reducing contention with concurrent reads/writes.
3. **Partial progress tracking** — The `MarkForReprocessResult` returns `failedBatches` and `failedCount` alongside `updatedCount`, which is only meaningful with independent batch transactions.
4. **Cursor advancement** (line 294) — The cursor moves forward past failed batches to avoid infinite retries, which only makes sense when batches are independent.

Wrapping everything in a single transaction would mean: one failure rolls back *all* work, locks are held longer, and the partial-failure reporting becomes pointless.

If you still want a single-transaction version (e.g., because you need all-or-nothing semantics for your use case), here's what it would look like — but be aware of the trade-offs:

```java
@Override
@Transactional
public MarkForReprocessResult markAllForReprocess() {
    int totalUpdated = 0;
    String runResult = "success";
    Timer.Sample runSample = Timer.start(meterRegistry);

    try {
        int effectiveBatchSize = Math.max(1, markInitBatchSize);
        Pageable pageable = PageRequest.of(0, effectiveBatchSize);

        long lastId = Long.MIN_VALUE;
        while (true) {
            List<Long> batchIds = entityRepository.findTopNIdsNeedMarkForReprocess(
                    lastId, ReprocessState.INIT, pageable);
            if (batchIds.isEmpty()) {
                break;
            }

            Integer updated = entityRepository.markBatchForReprocess(batchIds, ReprocessState.INIT);
            totalUpdated += (updated == null ? 0 : updated);

            lastId = batchIds.get(batchIds.size() - 1);
            if (batchIds.size() < effectiveBatchSize) {
                break;
            }
        }

        return MarkForReprocessResult.builder()
                .updatedCount(totalUpdated)
                .failedBatches(0)
                .failedCount(0)
                .build();
    } catch (Exception ex) {
        runResult = "failed";
        log.error("markAllForReprocess: failed, updatedCount={}", totalUpdated, ex);
        throw ex;
    } finally {
        markInitLastFailedBatches.set(0);
        runSample.stop(markInitRunTimeCost(runResult));
    }
}
```

Key differences: `@Transactional` on the method replaces the manual `TransactionTemplate`, no per-batch try/catch (any failure aborts everything), and `failedBatches`/`failedCount` are always 0 since partial failure is no longer possible.

**My recommendation**: Keep the existing batched design unless you have a specific requirement for all-or-nothing consistency.
