### `DeviceTypeChangedHandler` — what it does

**Trigger**: `CachedExistingEntityHandler` detects that the cached entity's type (e.g. `HOST`) differs from the incoming request's type (e.g. `USER_HOST_BINDING`) at line 53.

**Flow** (6 steps):

1. **Acquire process lock** — type change is a mutating operation, so it must be exclusive
2. **Re-read entity from DB** — avoids acting on stale cached data
3. **Rebuild fingerprint** — the attribute fingerprint changes when the entity type changes (different attributes are relevant)
4. **Update entity in DB** — sets new type, new fingerprint, resets processing stage to `PENDING`, clears error state. For `USER_HOST_BINDING`, populates `userId`/`userIdentity`
5. **Update cache** — writes the entity ID under the new exist-key so subsequent requests hit cache correctly
6. **Re-run state machine** — starts from `PENDING` so the entity goes through label → policy → enforcement with the new type

**Integration point** in `CachedExistingEntityHandler` (line 52-57):
```java
// device-type change detection: e.g. HOST → USER_HOST_BINDING
if (!e.getType().name().equals(req.getEntityType())) {
    return deviceTypeChangedHandler.handle(req, e);
}
```

This check runs **after** the stale-request version check (so we don't process outdated type changes) and **before** the fingerprint comparison (which is irrelevant when the type itself changed).
