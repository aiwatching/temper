I'd push back on this change. The current design is **intentionally** using separate transactions per batch, and the code comments explain why (lines 266-270):

> "one transaction per update batch to avoid a huge long transaction"
> "if one batch fails, only that batch rolls back and already committed batches stay committed"

This is a deliberate architectural choice with clear benefits:

1. **Avoids long-running transactions** — a single transaction over all entities would hold locks for the entire duration, blocking other writers
2. **Partial failure resilience** — if batch N fails, batches 1..N-1 are already committed. A single transaction would roll back *everything*
3. **Bounded memory/lock footprint** — each batch only locks `markInitBatchSize` rows at a time

Wrapping the entire operation in one transaction would:
- Hold a large number of row locks for the full duration
- Risk lock timeouts or deadlocks under concurrent writes
- Lose all progress if any single batch fails
- Potentially hit transaction log size limits on large tables

The current batched approach is the right pattern for bulk updates. Unless you have a specific consistency requirement where all-or-nothing semantics are needed (e.g., all entities must be marked atomically or none at all), I'd recommend keeping the current implementation as-is.
