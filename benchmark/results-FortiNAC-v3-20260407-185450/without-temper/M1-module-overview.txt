## core_business_v2/ingestion Module Architecture

### Overview

The ingestion module is an **event-driven entity processing pipeline** built on Spring Boot + Redis Streams. It receives network entity data (hosts, users, adapters), deduplicates via fingerprinting, and processes each entity through a 4-stage state machine.

---

### REST Endpoints

| Method | Path | Purpose |
|--------|------|---------|
| `POST` | `/api/v1/ingestion/events` | Main entry — ingests an entity event, returns async result |
| `GET` | `/api/v1/ingestion/health-check` | Health probe |
| `POST` | `/api/v1/entities/query` | Query entities by stage/code with pagination |
| `GET` | `/api/v1/entities/{id}` | Get entity by ID |
| `GET` | `/api/v1/entities/{id}/display-status` | Human-readable processing status |
| `POST` | `/api/v1/fingerprint/calculate` | Calculate attribute fingerprint (MD5) |
| `POST` | `/api/v1/policy/apply-existing-entities` | Mark all entities for policy re-evaluation |

---

### Request Processing Flow

```
REST POST /ingestion/events
  → IngestionController: generate requestId, register CompletableFuture, publish to Redis Stream
    → IngestionInternalConsumer: deserialize, call orchestrator
      → IngestionOrchestrator (3-tier lookup):
          1. Redis cache hit? → CachedExistingEntityHandler
          2. Acquire create-lock → DB lookup → DbExistingEntityHandler
          3. No DB match → NewEntityHandler
```

**For existing entities**, the orchestrator computes a fingerprint and selects one of 4 handlers via `FingerprintHandlerRegistry`:

| Fingerprint | Lock Acquired | Handler | Action |
|-------------|--------------|---------|--------|
| Same | Yes | SameAttributesWithLockAcquiredHandler | No-op, return entity |
| Same | No | SameAttributesWithLockNotAcquiredHandler | Spin-wait for current processing to finish |
| Different | Yes | DifferentAttributesWithLockAcquiredHandler | Reset to PENDING, restart state machine |
| Different | No | DifferentAttributesWithLockNotAcquiredHandler | **Request merge**: interrupt running SM, acquire lock, restart |

A special `DeviceTypeChangedHandler` handles type escalation (HOST → USER_HOST_BINDING).

---

### State Machine (`EntityStateMachine`)

4 stages, executed sequentially in a loop:

```
PENDING → LabelService.label() → LABELED
LABELED → PolicyService.evaluate() → EVALUATED
EVALUATED → EnforcementService.enforce() → ENFORCED (terminal)
```

Key design decisions:
- **Each stage persists independently** via `@Transactional(REQUIRES_NEW)` — crash recovery resumes from last saved stage
- **Per-stage retry**: up to 3 attempts with exponential backoff + jitter
- **Interruptible**: checks `entity.interrupted` before each stage (for request merge)
- **Failpoint injection**: chaos testing hooks before/after each service call
- Tracks `ingestion_state_machine_seconds{result, finalStage}` metrics

---

### Key Classes by Layer

**Controllers**: `IngestionController`, `EntityController`, `FingerprintController`, `PolicyController`

**Consumer/Orchestration**: `IngestionInternalConsumer`, `IngestionExternalConsumer`, `IngestionOrchestrator`

**Handlers** (strategy pattern): 4 fingerprint handlers + `NewEntityHandler` + `DeviceTypeChangedHandler`

**State Machine**: `EntityStateMachine`, `EntityStageTransactionRunner`

**Stage Services**: `LabelService` (rule engine), `PolicyService` (evaluation), `EnforcementService` (actions)

**Domain**: `Entity` (main JPA entity with processing state), `HostRecord`, `UserRecord`, `DynamicClient`

**Infrastructure**: `EntityCache` (Redis), `RedisDelayQueueUtil` (exponential backoff requeue), `IngestionCallbackRegistry` (async REST → future), `EntityProcessingRegistry` (active processing tracker)

**Recovery**: `EntityRecoveryService`, `StuckEntityRecoveryJob`, `PolicyReprocessJob`

---

### Concurrency Model

- **Redis distributed locks**: create-lock (10s TTL) prevents duplicate entity creation; process-lock (30s TTL) serializes state machine execution per entity
- **Lock contention** → delay queue with exponential backoff (2s → 32s)
- **Request merge**: when a new request arrives with different attributes while the entity is mid-processing, the running state machine is interrupted and restarted with merged context
- **Async REST**: controller registers a `CompletableFuture`, returns `Mono`; consumer completes the future when processing finishes (configurable 60s timeout)
