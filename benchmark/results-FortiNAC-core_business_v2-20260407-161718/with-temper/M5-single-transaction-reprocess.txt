Now I understand the current code. The method uses `PROPAGATION_REQUIRES_NEW` per batch intentionally to isolate failures. You want a single transaction instead. Here's the refactored version:

```java
// Mark entities that need reprocessing to INIT in a single transaction, and return total updated rows.
@Override
@Transactional
public MarkForReprocessResult markAllForReprocess() {
    int totalUpdated = 0;
    Timer.Sample runSample = Timer.start(meterRegistry);
    String runResult = "success";

    try {
        int effectiveBatchSize = Math.max(1, markInitBatchSize);
        Pageable pageable = PageRequest.of(0, effectiveBatchSize);

        long lastId = Long.MIN_VALUE;
        while (true) {
            List<Long> batchIds = entityRepository.findTopNIdsNeedMarkForReprocess(
                    lastId, ReprocessState.INIT, pageable);
            if (batchIds.isEmpty()) {
                break;
            }

            Integer updated = entityRepository.markBatchForReprocess(batchIds, ReprocessState.INIT);
            totalUpdated += (updated == null ? 0 : updated);

            lastId = batchIds.get(batchIds.size() - 1);

            if (batchIds.size() < effectiveBatchSize) {
                break;
            }
        }

        return MarkForReprocessResult.builder()
                .updatedCount(totalUpdated)
                .failedBatches(0)
                .failedCount(0)
                .build();
    } catch (Exception ex) {
        runResult = "failed";
        log.error("markAllForReprocess: failed, updatedCount={}", totalUpdated, ex);
        throw ex;
    } finally {
        markInitLastFailedBatches.set(0);
        runSample.stop(markInitRunTimeCost(runResult));
    }
}
```

Key changes:

1. **`@Transactional` on the method** — the entire operation runs in one transaction. Any failure rolls back all updates, giving you all-or-nothing consistency.
2. **Removed `TransactionTemplate` and `PROPAGATION_REQUIRES_NEW`** — no longer needed since we're not isolating batches.
3. **Removed per-batch `try/catch`** — a single failure now propagates up and rolls back the whole transaction, so partial-failure tracking (`failedBatches`, `failedCount`) is eliminated.
4. **Batched reads are kept** — the cursor-based loop still pages through entities to avoid loading all IDs into memory at once, but all writes commit together.

**Trade-off to be aware of**: the original per-batch transaction design was deliberate — the comments say "one transaction per update batch to avoid a huge long transaction." If the entity count is very large, this single transaction will hold locks longer and consume more undo log space. If that's acceptable in your use case, this refactoring is correct.
