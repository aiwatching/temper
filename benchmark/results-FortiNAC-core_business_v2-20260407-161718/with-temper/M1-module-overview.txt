## core_business_v2 Ingestion Module

### Overview

The ingestion module is a Spring Boot service that processes entity ingestion requests (hosts, user-host bindings) through a multi-stage pipeline: **Label → Policy → Enforcement**. It uses Redis streams for async message processing, distributed locks for concurrency, and a fingerprint-based deduplication strategy.

---

### REST Endpoints

| Method | Path | Purpose |
|--------|------|---------|
| `POST` | `/api/v1/ingestion/events` | Main ingestion entry point — accepts `IngestionRequest`, publishes to Redis stream, waits up to 60s for result |
| `POST` | `/api/v1/entities/query` | Query failed entities with pagination (`EntityQueryDTO`) |
| `GET` | `/api/v1/entities/{id}` | Get entity by ID |
| `GET` | `/api/v1/entities/{id}/display-status` | Get display status (init/running/enforced/failed) |
| `POST` | `/api/v1/fingerprint/calculate` | Calculate fingerprint for given entity attributes |
| `POST` | `/api/v1/policy/apply-existing-entities` | Mark all entities for reprocessing |

### Request Processing Flow

```
Client POST /api/v1/ingestion/events
  │
  ▼
IngestionController (127 lines)
  ├── Generate requestId + traceId
  ├── Register CompletableFuture in CallbackRegistry (60s timeout)
  ├── Publish IngestionRequest to Redis INGESTION_INTERNAL_STREAM
  └── Wait for future completion on nettyOffloadExecutor
        │
        ▼
IngestionInternalConsumer (181 lines) — Redis stream consumer
  │
  ▼
IngestionOrchestrator.process(req) (107 lines)
  ├── 1. Cache lookup: EntityCache.getCachedEntityId(key)
  │     ├── HIT  → CachedExistingEntityHandler
  │     └── MISS → Acquire CREATE_LOCK
  │
  ├── 2. DB lookup (under lock)
  │     ├── Found → DbExistingEntityHandler → CachedExistingEntityHandler
  │     └── Not found → NewEntityHandler
  │
  ▼
Handler Selection (fingerprint comparison + lock state):
  ┌─────────────────────┬────────────────┬──────────────────────┐
  │                     │ Lock Acquired  │ Lock Not Acquired    │
  ├─────────────────────┼────────────────┼──────────────────────┤
  │ Same Fingerprint    │ Return as-is   │ Poll until done      │
  │ Different Fingerprint│ Reset + run SM │ Interrupt + retry    │
  │                     │                │ lock + reset + run SM│
  └─────────────────────┴────────────────┴──────────────────────┘
  │
  ▼
EntityStateMachine.start() (336 lines)
  │
  PENDING ──[LabelService]──→ LABELED
  LABELED ──[PolicyService]──→ EVALUATED
  EVALUATED ──[EnforcementService]──→ ENFORCED
  │
  Each stage: up to 3 retries with exponential backoff (1s×attempt + jitter)
  Retryable vs non-retryable codes determine whether to retry or fail
  DB updated after each stage transition
  │
  ▼
Consumer completes the CallbackRegistry future → Controller returns ResultDTO to client
```

### Key Classes & Responsibilities

**Controllers:**
- `IngestionController` — async ingest via Redis stream + callback future
- `EntityController` — entity queries and status display
- `PolicyController` — trigger bulk reprocessing

**Orchestration:**
- `IngestionOrchestrator` — routes requests: cache hit → existing handler, cache miss → lock → DB lookup → new/existing handler
- `IngestionInternalConsumer` / `IngestionExternalConsumer` — Redis stream consumers

**Handlers (4 fingerprint cases):**
- `SameAttributesWithLockAcquiredHandler` — no-op, entity unchanged
- `SameAttributesWithLockNotAcquiredHandler` — poll until processing finishes
- `DifferentAttributesWithLockAcquiredHandler` — reset entity, run state machine
- `DifferentAttributesWithLockNotAcquiredHandler` — interrupt running SM, acquire lock, reset, run SM

**State Machine:**
- `EntityStateMachine` — drives PENDING→LABELED→EVALUATED→ENFORCED with retry logic
- `EntityStageTransactionRunner` — wraps each stage call in `@Transactional`

**Services (stage executors):**
- `LabelService` — calls RuleEngineService for labeling
- `PolicyService` — policy evaluation (currently placeholder)
- `EnforcementService` — enforcement (currently placeholder)

**Utilities:**
- `EntityCache` — Redis-backed cache for entity IDs (Redisson `RBucket`)
- `EntityUtil` — SHA256 fingerprint from 40+ host fields, 34 user fields, 2 dynamic client fields
- `RedisUtil` — lock key generation per entity type

**Background Jobs:**
- `PolicyReprocessJob` (364 lines) — batch reprocessing of entities marked INIT
- `StuckEntityRecoveryJob` (229 lines) — recovers entities stuck in intermediate states

### Architectural Patterns

1. **Async request/response** — REST call → Redis stream → consumer processes → completes `CompletableFuture` → REST returns
2. **Fingerprint deduplication** — SHA256 of entity attributes determines if reprocessing is needed
3. **Two-tier locking** — `CREATE_LOCK` (entity creation, fast-fail) + `PROCESS_LOCK` (state machine execution, supports spin-retry & interruption)
4. **Graceful degradation** — cache miss falls through to DB; cache write failure degrades to DB-only mode
5. **Delay queue** — failed requests requeued to Redis sorted set with exponential backoff
