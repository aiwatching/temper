So `spring-boot-starter-web` is only referenced as an exclusion, and `spring-boot-starter-webflux` is the actual dependency. This is a **WebFlux** application running on Netty — that's why the Mono + `subscribeOn` pattern exists: to avoid blocking Netty's event loop.

**Important consideration**: Since this is a WebFlux app on Netty (not Tomcat), simply returning a blocking `ResultDTO<?>` from the controller would block the Netty event loop thread, which is exactly what the current code is designed to avoid. The `subscribeOn(Schedulers.fromExecutor(...))` offloads the blocking work to a separate thread pool.

If you want to simplify this, there are two paths:

1. **Switch to Spring MVC** (add `spring-boot-starter-web`, remove `webflux`) — then blocking controllers are fine because Tomcat uses thread-per-request.
2. **Keep WebFlux but simplify the Mono chain** — still need Mono but can flatten the code.
