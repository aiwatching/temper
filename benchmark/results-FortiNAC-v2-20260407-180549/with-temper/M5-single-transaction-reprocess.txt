I need to push back on this. The current implementation is **intentionally designed** with separate transactions per batch — the comments explicitly explain why (lines 266-270):

> "one transaction per update batch to avoid a huge long transaction"
> "if one batch fails, only that batch rolls back and already committed batches stay committed"

This is the correct pattern for bulk updates. Wrapping the entire operation in a single transaction would:

1. **Hold a long-running transaction** locking many rows for the entire duration, blocking other writers
2. **Lose partial progress** — if the 50th batch fails, all previous batches roll back too
3. **Risk transaction timeout** on large datasets
4. **Increase memory pressure** — the DB must track undo logs for all rows until commit

The per-batch approach with `PROPAGATION_REQUIRES_NEW` gives you:
- **Fault isolation**: one bad batch doesn't nuke the whole run
- **Shorter lock windows**: each batch locks rows briefly then releases
- **Observable progress**: `failedBatches`/`failedCount` metrics are meaningful because committed batches are real

The method already returns `MarkForReprocessResult` with `failedBatches` and `failedCount`, and the caller can decide whether a `partial_failure` needs retry. This is a deliberate resilience pattern, not an oversight. I'd leave it as-is.
