## Diagnosis: Entities Stuck in LABELED State

There is a **bug in `EntityStateMachine.processStage()`** at lines 208-212 that causes the LABELED→EVALUATED stuckness.

### The Bug

```java
// EntityStateMachine.java:208-212 — runs UNCONDITIONALLY after the retry loop
dto.setProcessingStage(nextStage);                    // always advances
boolean dbSuccess = updateDBEntityProcessingState(dto);
return serviceSuccess && dbSuccess;
```

When `PolicyService.evaluate()` **fails**, `serviceSuccess` is `false`, but the code **still writes `stage=EVALUATED`** to the DB. The method returns `false`, so the state machine loop exits — but the DB now has `stage=EVALUATED, code=non-zero`. Those entities aren't stuck in LABELED, they're incorrectly in EVALUATED with a failure code.

### What actually causes LABELED + code=0 (truly stuck)

The `@Transactional` annotation on `EntityStageTransactionRunner.runPolicy()` (line 24-26) is the culprit. If `PolicyService.evaluate()` triggers a **Spring transaction rollback** (e.g., DB constraint violation within the transaction), the `StageExecutor` catches the exception and returns `code=-1` (SYSTEM_ERROR). But the `@Transactional` proxy **still marks the transaction as rollback-only**.

Here's the sequence:
1. PENDING → LABELED succeeds, DB has `stage=LABELED, code=0`
2. `txRunner.runPolicy(dto)` starts a new transaction
3. Inside the transaction, something causes a rollback-only (e.g., lazy-load triggers a stale entity read, or a transactional side-effect fails)
4. `StageExecutor` catches the exception, returns `{code: -1}`
5. State machine retries 3 times — all fail the same way
6. `handleServiceFinalFailure` sets `code=-1` on the DTO
7. Line 209: `dto.setProcessingStage("EVALUATED")`
8. Line 210: `updateDBEntityProcessingState()` — this runs **outside** the `@Transactional` method, so it tries a direct DB update. If the entity's `updatedAt` optimistic lock check fails (stale), `rows=0` → returns false
9. **DB still has `stage=LABELED, code=0`** from step 1

The recovery job picks these up after 80s, but if the same condition persists (e.g., corrupted entity data causing repeated transaction rollbacks), recovery also fails, creating a **permanent stuck loop**.

### How to Fix

**Fix 1 — Don't advance stage on failure** (the primary bug):

```java
// EntityStateMachine.java:processStage() — only advance stage on success
if (serviceSuccess) {
    dto.setProcessingStage(nextStage);
} 
// Always persist current state (success or failure) to DB
boolean dbSuccess = updateDBEntityProcessingState(dto);
return serviceSuccess && dbSuccess;
```

This ensures failed entities stay in their current stage with a non-zero code, making the failure visible and preventing the stage/code mismatch.

**Fix 2 — Add a "stuck count" or dead-letter mechanism**:

In `EntityRecoveryService.recoverIfStuck()`, track how many times an entity has been recovered. After N attempts (e.g., 5), mark it as permanently failed with a specific error code rather than retrying infinitely:

```java
if (fresh.getRecoveryCount() >= MAX_RECOVERY_ATTEMPTS) {
    entityService.markPermanentlyFailed(entityId, "exceeded max recovery attempts");
    return new RecoveryResult(true, true, false);
}
entityService.incrementRecoveryCount(entityId);
```

**Fix 3 — Separate transaction boundaries**:

The `@Transactional` on `runPolicy()` wraps the entire service call. If you only need transactional behavior for DB writes within the policy evaluation, narrow the transaction scope so that transactional failures don't prevent the state machine from recording the failure.
