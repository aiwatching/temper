Before I show the refactored code, I want to flag something: the current design is **intentional**. The comments on lines 266-270 explain why:

> "one transaction per update batch to avoid a huge long transaction"  
> "if one batch fails, only that batch rolls back and already committed batches stay committed"

This is a resilience pattern — partial progress is preserved if a batch fails. Wrapping everything in a single transaction gives you all-or-nothing consistency, but you lose:

1. **Partial progress** on failure — one bad row rolls back everything
2. **Bounded lock duration** — a single large transaction holds row locks longer, increasing contention
3. **The `failedBatches`/`partial_failure` tracking** becomes meaningless (any failure rolls back all)

If you still want single-transaction consistency, here's the refactored version:

```java
// Mark entities that need reprocessing to INIT in a single transaction, and return total updated rows.
@Override
@Transactional
public MarkForReprocessResult markAllForReprocess() {
    int totalUpdated = 0;
    String runResult = "success";
    Timer.Sample runSample = Timer.start(meterRegistry);

    try {
        int effectiveBatchSize = Math.max(1, markInitBatchSize);
        Pageable pageable = PageRequest.of(0, effectiveBatchSize);

        long lastId = Long.MIN_VALUE;
        while (true) {
            List<Long> batchIds = entityRepository.findTopNIdsNeedMarkForReprocess(
                    lastId, ReprocessState.INIT, pageable);
            if (batchIds.isEmpty()) {
                break;
            }

            Integer updated = entityRepository.markBatchForReprocess(batchIds, ReprocessState.INIT);
            totalUpdated += updated == null ? 0 : updated;

            lastId = batchIds.get(batchIds.size() - 1);

            if (batchIds.size() < effectiveBatchSize) {
                break;
            }
        }

        return MarkForReprocessResult.builder()
                .updatedCount(totalUpdated)
                .failedBatches(0)
                .failedCount(0)
                .build();
    } catch (Exception ex) {
        runResult = "failed";
        log.error("markAllForReprocess: failed, updatedCount={}", totalUpdated, ex);
        throw ex;
    } finally {
        markInitLastFailedBatches.set(0);
        runSample.stop(markInitRunTimeCost(runResult));
    }
}
```

Key changes:
- `@Transactional` on the method — single transaction, any exception rolls back everything
- Removed `TransactionTemplate`, `failedBatches` tracking, and per-batch try/catch (no longer meaningful)
- Batched reads are retained for cursor-based pagination (still efficient for large datasets)
- `MarkForReprocessResult.failedBatches` is always 0 — failure throws, triggering full rollback

The batched **reads** (keyset pagination) still prevent loading all IDs into memory at once. Only the **transaction boundary** changed.
