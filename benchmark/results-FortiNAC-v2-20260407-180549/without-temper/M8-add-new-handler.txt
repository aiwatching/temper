## Summary

**`DeviceTypeChangedHandler`** (already existed at line 37) handles the actual type-change logic:
1. Acquires process lock
2. Re-reads entity from DB to avoid stale-cache races
3. Rebuilds fingerprint for the new type
4. Updates entity: type, fingerprint, resets processing stage to `PENDING`
5. For `USER_HOST_BINDING`, populates `userId`/`userIdentity`
6. Updates cache
7. Re-runs the state machine from `PENDING`

**`CachedExistingEntityHandler`** (the wiring point) — I added:
- **Line 30**: Injected `DeviceTypeChangedHandler` via constructor
- **Lines 53-60**: After the version/staleness check but **before** fingerprint comparison, checks if `entity.type != req.entityType`. If they differ, delegates directly to `DeviceTypeChangedHandler`, bypassing the fingerprint logic entirely (since a type change always requires reprocessing regardless of attribute changes).

**Flow**: `Orchestrator` → cache hit → `CachedExistingEntityHandler.handle()` → version check → **type-change check (new)** → fingerprint comparison → `FingerprintHandler`

The type check is placed after the version check so stale requests are still discarded, but before fingerprint comparison so type changes short-circuit the normal same/different-attributes flow.
