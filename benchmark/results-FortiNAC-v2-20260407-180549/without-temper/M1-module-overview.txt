## core_business_v2/ingestion Module Overview

### What It Does
An **async event-driven entity ingestion pipeline** — receives entity events (hosts, user-host bindings) via REST, processes them through a 4-stage state machine (label → evaluate → enforce), with Redis-based distributed locking, fingerprint deduplication, and request interruption support.

---

### REST Endpoints

| Method | Path | Purpose |
|--------|------|---------|
| **POST** | `/api/v1/ingestion/events` | Main entry — publishes to Redis stream, waits 60s for async result |
| **GET** | `/api/v1/ingestion/health-check` | Returns "OK" |
| **POST** | `/api/v1/entities/query` | Query failed entities with filters + pagination |
| **GET** | `/api/v1/entities/{id}` | Get entity by ID |
| **GET** | `/api/v1/entities/{id}/display-status` | Display-friendly state mapping |
| **POST** | `/api/v1/policy/apply-existing-entities` | Mark all entities for policy reprocessing |
| **POST** | `/api/v1/fingerprint/calculate` | Calculate fingerprint hash for given entity |

---

### Request Processing Flow

```
POST /api/v1/ingestion/events
        │
        ▼
IngestionController — generates requestId, registers CompletableFuture callback, publishes to Redis stream
        │
        ▼
IngestionInternalConsumer (16 consumers) — reads from stream, calls IngestionOrchestrator
        │
        ▼
IngestionOrchestrator — 3-way branch:
  ├─ Cache HIT → CachedExistingEntityHandler
  ├─ Cache MISS + lock acquired → DB lookup → NewEntityHandler or DbExistingEntityHandler
  └─ Cache MISS + lock failed → requeue with exponential backoff (2-32s, max 5 retries)
        │
        ▼
FingerprintHandlerRegistry — dispatches on (sameFingerprint, lockAcquired):
  ├─ SAME + LOCK → return entity as-is (idempotent)
  ├─ SAME + NO_LOCK → poll DB 60s waiting for completion
  ├─ DIFF + LOCK → reset to PENDING, restart state machine
  └─ DIFF + NO_LOCK → interrupt running SM, acquire lock, reset & restart
        │
        ▼
EntityStateMachine.start() — loops through stages:
  PENDING → LabelService.label()
  LABELED → PolicyService.evaluate()
  EVALUATED → EnforcementService.enforce()
  ENFORCED (terminal)
  Each stage: 3 retries with backoff, DB persist on transition
        │
        ▼
Callback completes → REST response returned to caller
```

### Key Classes & Responsibilities

**Controllers:** `IngestionController`, `EntityController`, `PolicyController`, `FingerprintController`

**Consumer Layer:**
- **IngestionInternalConsumer** — Redis stream consumer (16 instances), ACK-based, drives the pipeline
- **IngestionExternalConsumer** — Ingests from external stream (8 instances), validates, republishes internally
- **IngestionOrchestrator** — Cache/lock decision tree, routes to appropriate handler

**Handlers (fingerprint-based dispatch):**
- **NewEntityHandler** — Creates entity, builds fingerprint, acquires PROCESS_LOCK, starts SM
- **CachedExistingEntityHandler / DbExistingEntityHandler** — Fingerprint comparison → FingerprintHandlerRegistry
- **DeviceTypeChangedHandler** — Handles HOST → USER_HOST_BINDING promotion
- **SameAttributes[With/Without]LockHandler**, **DifferentAttributes[With/Without]LockHandler** — 4 concrete strategies

**State Machine:**
- **EntityStateMachine** — Stage loop with retry, failpoint injection, metrics
- **EntityStageTransactionRunner** — Wraps each stage call in a transaction

**Services:**
- **LabelService** — Calls classification rule engine
- **PolicyService** — Policy evaluation (currently mock)
- **EnforcementService** — Enforcement action (currently mock)
- **EntityServiceImpl** — CRUD, batch operations, reprocess marking
- **FingerprintServiceImpl** — Attribute hash computation

**Infrastructure:**
- **IngestionCallbackRegistry** — requestId → CompletableFuture mapping (bridges async consumer back to REST)
- **EntityCache** — Redis-based entity ID cache
- **DistributedLock** — Redis CREATE_LOCK and PROCESS_LOCK

**Jobs:**
- **PolicyReprocessJob** — Scans entities marked INIT, re-runs SM (every 30s + event-driven)
- **StuckEntityRecoveryJob** — Recovers entities stuck >80s in non-terminal state

---

### Key Design Patterns

1. **Callback Registry** — Decouples synchronous REST from async stream processing
2. **Distributed Locking** — Redis-based CREATE_LOCK / PROCESS_LOCK with TTL
3. **Fingerprint Deduplication** — Attribute hash comparison to skip redundant processing
4. **Request Interruption** — Newer requests interrupt stale in-flight processing
5. **State Machine with Retry** — 4-stage pipeline, 3 retries per stage, exponential backoff
6. **Handler Registry** — Enum-based dispatch on `(sameFingerprint, lockAcquired)` → 4 strategies
